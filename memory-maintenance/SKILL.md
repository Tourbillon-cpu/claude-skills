# 记忆文件维护手册

> 对抗熵增 —— 让记忆库永远保持清爽

## 适用场景

- 用户说"清理一下记忆"、"整理记忆文件"
- 用户说"该对抗一下熵增了"
- 记忆文件内容过时需要批量更新
- 重复内容、冗余信息需要清理
- 新文件需要创建，需要符合规范
- 生命周期标记需要更新（temporary 过期 → 删除）

---

## 核心理念

### 两个维度分类

| 维度 | 分类 | 说明 |
|------|------|------|
| **性质** | `user/` / `project/` / `reference/` | 给人看：这个文件记录的是什么类型的内容 |
| **时间** | `permanent` / `temporary` | 给维护看：这个文件用多久，过期就删 |

一个文件同时属于两种分类——**目录决定性质，生命周期标记决定时长。**

### 三大原则

1. **每个文件只讲一件事** → 内容交叉了就该拆分或合并
2. **不重复** → 同一信息只存在一个文件里，其他地方引用它
3. **过期就删** → temporary 的任务结束了，文件就该删除

## 目录结构规范

```
memory/
├── MEMORY.md                ← 索引（每行一条，[常驻] / [临时] 前缀）
│
├── user/                    ← 关于"你这个人"：身份、环境、偏好（永久）
│   ├── about-me.md          ← 个人基本信息
│   ├── work-preferences.md  ← 工作偏好与助手行为准则
│   └── dev-setup.md         ← 硬件设备与跨设备同步方案
│
├── project/                 ← 阶段性任务（做完即删）
│   ├── my-study.md          ← 期末复习进度
│   └── current-lab-work.md  ← 实验数据进度
│
└── reference/               ← 技术备忘（永久）
    ├── origin-api.md        ← Origin API 参考
    └── ocr-toolchain.md     ← OCR 工具链
```

### 目录创建规则

- `user/` — 个人身份、偏好、设备环境
- `project/` — 有明确结束时间的任务
- `reference/` — 纯技术参考、解决方案备忘

> **注意：** `dev-setup.md` 虽然记录设备和环境，但它属于"你这个人"的信息，故归入 `user/`，不归 `reference/`。

## 文件命名规则

- 全英文小写 kebab-case（短横线连接）
- 长度适中，能概括内容
- 不加特殊符号（`[]` `()` 等——shell 中可能引发通配符解析）

```
✅ 正确: about-me.md, work-preferences.md, origin-api.md, ocr-toolchain.md
❌ 错误: 关于我.md, work_preferences.md, originAPI.md, [常驻]about-me.md
```

---

## 生命周期标记规范

每个文件在 frontmatter 之后的正文第一行，添加生命周期标记：

```markdown
---

> **生命周期：** permanent

## 正文标题...
```

| 标记 | 含义 | 清理策略 |
|------|------|---------|
| `permanent` | 常驻，不主动删除 | 除非内容彻底失效，否则保留 |
| `temporary（考完后删除）` | 考试结束后删除 | 考试全完了就删 |
| `temporary（实验报告交完后删除）` | 实验报告提交后删除 | 报告交了、无可更新内容就删 |

---

## Frontmatter 规范

```yaml
---
name: short-kebab-case-name
description: 一句话说明文件内容
metadata:
  type: user | project | reference
---
```

| 字段 | 要求 |
|------|------|
| `name` | 小写 kebab-case，与文件名主体一致 |
| `description` | 一句话，让人/我一眼知道要不要读这个文件 |
| `metadata.type` | 与所在目录对应 |

## MEMORY.md 索引规范

### 格式

```
- [常驻] [file-name](path/file-name.md) — 一句话描述
- [临时] [file-name](path/file-name.md) — 一句话描述
```

### 更新要点

1. **创建新文件时**：同时在 MEMORY.md 末尾加一行
2. **删除文件时**：同时删掉 MEMORY.md 对应行
3. **移动文件时**：更新 MEMORY.md 中的路径
4. **前缀 `[常驻]` / `[临时]` 必须标注**，方便新会话的我一眼区分

## 内容撰写规范

### 跨文件链接

在同一知识库内引用其他记忆文件：

```
[[about-me]]
[[dev-setup]]
```

链接使用 frontmatter 中的 `name:` 字段值，不依赖文件路径。

### 内容交叉处理

当发现两个文件有相同内容时：

1. **判断哪个文件更合适保留**（按"每件事只讲一次"原则）
2. **被保留的文件补充完整信息**
3. **被删除的文件删掉，重复内容从另一文件移除**
4. **删除的文件在 MEMORY.md 中同步移除**

### 不要放什么

- ❌ 不需要把 git 历史、代码结构、过去的修复记录写进记忆——这些在仓库里
- ❌ 一次性修复方案（如 MathType 修复），修完就删
- ❌ 临时性的、只在本会话有用的信息

## 定期清理策略

| 触发器 | 行动 |
|--------|------|
| 用户说"清理记忆" | 全面审查所有文件 |
| temporary 任务结束 | 立即删除对应文件 |
| 发现内容重复 | 合并/拆分，保持"每件只讲一次" |
| 信息过时 | 更新内容，必要时改为 permanent 或删除 |

### 清理检查清单

- [ ] 所有 `project/` 下的文件，任务是否已经结束？
- [ ] 有内容重复吗？同一信息是否在多处出现？
- [ ] 信息是否过时？日期、版本号、状态是否需要更新？
- [ ] 是否存在一次性修复内容，已经搞定可以删？
- [ ] MEMORY.md 的路径和文件实际存在的是否一致？
- [ ] 生命周期标记正确吗？（temporary 的是否该删了？）

---

## 与 work-preferences.md 的关联

`user/work-preferences.md` 记录了三条工作偏好 + 五条工作流规则 + 可用工具索引。在整理记忆时注意：

1. **工作流规则（方案先于执行、你确认才动、逐节点汇报、跑偏退回、最终验收）**——整理记忆本身也是"处理文件/文档"的工作，同样适用
2. **知识库规则**——不要往 KnowledgeStorage 里塞无关的大文件；内容严格按课件原文
3. **清理完成后**——在 MEMORY.md 和 work-preferences.md 的"可用工具"表中同步更新路径和引用

## 示例：一次完整的清理流程

```
1. 用户说"该对抗一下熵增了"
   ↓
2. 读取 MEMORY.md，遍历所有文件
   ↓
3. 检查 project/ 下是否有 expired temporary 文件 → 删除
   ↓
4. 检查是否有内容重复（如 A 和 B 都在说同一件事）→ 合并
   ↓
5. 检查是否有过时信息（日期、版本）→ 更新
   ↓
6. 检查 MEMORY.md 路径与实际是否一致 → 修正
   ↓
7. 汇报清理结果：删了什么、改了什么、新增了什么
   ↓
8. 用户验收
```
